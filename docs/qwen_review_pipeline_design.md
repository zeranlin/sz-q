# 适配 Qwen3.5-27B 的政府采购招标文件审查流水线设计

## 1. 目标

这套流水线面向政府采购招标文件的合规审查，重点解决以下问题：

- 单份文件很长，模型一次性阅读容易漏风险
- 小中型模型在长上下文下稳定性不足
- 直接让模型生成最终 Markdown，格式容易漂移
- 审查结果需要可回溯到原文行号和证据片段

设计目标不是“让模型像专家一样凭空阅读全文”，而是通过“规则先召回、模型做定性、程序统一输出”的方式，把 Qwen3.5-27B 的能力用在最稳的环节。

## 2. 总体架构

流水线分为 7 层：

1. 文档解析层
2. 行号标准化层
3. 规则召回层
4. 分类审查层
5. 结果去重与合并层
6. Markdown 渲染层
7. 调试与质检层

推荐的数据流如下：

```text
doc/docx/pdf
  -> 纯文本抽取
  -> 带行号文本
  -> 规则召回候选片段
  -> 按风险类别分桶
  -> LLM 分类审查（JSON）
  -> 去重/合并
  -> Markdown 输出
```

## 3. 为什么适合 Qwen3.5-27B

Qwen3.5-27B 的主要挑战通常不是“不会判断”，而是：

- 一次看太多内容时漏项
- 输出格式不稳定
- 容易对相似条款重复定性
- 法律依据容易散

因此要做以下约束：

- 不给整份全文，只给候选片段
- 不让模型自由发挥，按单一审查类型分批审
- 不直接输出 Markdown，先输出结构化 JSON
- 法律依据从预置法条池里选
- 最后用程序统一渲染

## 4. 审查类型拆分

推荐至少拆成以下 8 类。每类单独调用一次 LLM。

1. 评分因素关联性审查
2. 资格条件合理性审查
3. 业绩设置合理性审查
4. 证书认证设置合理性审查
5. 检测报告与证明材料审查
6. 样品/演示/讲解审查
7. 技术参数倾向性审查
8. 文件完整性与模板残留审查

如果后续扩展到更多品类，还可以增加：

- 医疗设备进口/授权审查
- 信息化项目知识产权与源代码审查
- 服务项目驻场、本地化、人员资质审查

## 5. 文档解析层

### 5.1 输入

- `.docx`
- `.doc`
- `.pdf`
- `.txt`

### 5.2 输出

每份文件生成一份标准化文本：

- 保留文件名
- 按行拆分
- 添加 1-based 行号
- 保存为中间产物，便于人工复核

示例：

```text
1  深圳中学西校区学生宿舍楼家具采购
2  招标文件信息
3  项目编号：
```

### 5.3 解析原则

- `.docx/.doc` 优先走系统转换工具，如 macOS 的 `textutil`
- `.pdf` 可后续补 OCR 或 PDF 抽取
- 不在解析阶段做“智能理解”
- 只做稳定、可追踪的文本提取

## 6. 规则召回层

这一层不做最终定性，只负责把“可能有风险”的片段从全文中捞出来。

### 6.1 召回目标

- 把一份几千行文件缩到几十个候选片段
- 片段要带起止行号和原文
- 优先高召回，允许一定冗余

### 6.2 推荐规则分类

#### A. 证书/认证类

关键词示例：

- `认证证书`
- `ISO`
- `体系认证`
- `五星级`
- `售后服务认证`
- `软件著作权`
- `履约能力`

#### B. 业绩类

- `同类业绩`
- `中标通知书`
- `验收报告`
- `发票`
- `合同关键页`

#### C. 样品/演示类

- `样品`
- `演示`
- `讲解`
- `优`
- `良`
- `中`
- `差`
- `美观`
- `精美`
- `质感`

#### D. 检测报告类

- `检测报告`
- `CMA`
- `CNAS`
- `原件备查`
- `委托单位`
- `查询截图`

#### E. 设备类

- `生产设备`
- `喷涂线`
- `开料锯`
- `CNC`
- `设备购买合同`
- `租赁合同`

#### F. 倾向性参数类

- `进口品牌`
- `优质品牌`
- `同一品牌`
- `唯一授权`
- `指定`
- `保税库`

#### G. 模板/未定稿类

- `示例`
- `xx`
- `20XX`
- `x分`
- `扣？分`
- `***设备`

### 6.3 片段切分原则

命中行不是直接送模型，而是截取一个局部窗口。

建议：

- 命中行前后各取 2-4 行
- 若相邻命中行距离很近，则合并成一个片段
- 单片段控制在 200-600 汉字更稳

## 7. 分类审查层

### 7.1 核心原则

同一轮 LLM 调用只做一件事：

- 只审一个风险类别
- 只看这一类召回片段
- 只输出结构化 JSON

### 7.2 不推荐的方式

```text
请审查这份招标文件所有问题。
```

这种方式对 Qwen3.5-27B 很不友好，容易：

- 遗漏高风险条款
- 把一般风险写成高风险
- 输出重复 findings
- 格式失控

### 7.3 推荐 Prompt 结构

每类 prompt 分四部分：

1. 角色
2. 审查目标
3. 输出 schema
4. 候选片段

内部输出统一为 JSON 数组，字段推荐如下：

```json
[
  {
    "title": "评分项与采购标的不相关",
    "risk_level": "高风险",
    "review_type": "评分因素关联性审查",
    "line_refs": ["第351行", "第484行"],
    "quotes": ["……", "……"],
    "reason": "……",
    "legal_basis": [
      "《中华人民共和国政府采购法实施条例》第二十条",
      "《政府采购货物和服务招标投标管理办法》（财政部令第87号）第五十五条"
    ]
  }
]
```

### 7.4 法条池约束

为降低小模型编造法条的概率，建议把可选法条固定在一个预置池里。

基础池建议：

- 《中华人民共和国政府采购法》
- 《中华人民共和国政府采购法实施条例》第二十条
- 《政府采购货物和服务招标投标管理办法》（财政部令第87号）第五十五条
- 《政府采购需求管理办法》（财库〔2021〕22号）

地区扩展池：

- 《深圳经济特区政府采购条例》
- 《深圳经济特区政府采购条例实施细则》

Prompt 中明确要求：

- 只能从给定法条池中选择
- 没有把握时少写，不要编造

## 8. 结果去重与合并层

模型输出通常会出现以下问题：

- 同一个问题被拆成两条
- 同一条款被不同类别重复命中
- 标题不同但本质相同

### 8.1 推荐去重策略

先做规则去重，再决定是否需要二次 LLM 归并。

规则去重依据：

- `review_type + line overlap`
- `title` 的关键词相似度
- `quotes` 的重合度

### 8.2 合并原则

如果两条 findings 同时满足以下条件，则可合并：

- 行号高度重叠
- 风险类型一致
- 风险理由属于同一主旨

合并后：

- 合并 `line_refs`
- 保留 2-4 条代表性 `quotes`
- 生成一条更完整的 `reason`

## 9. Markdown 渲染层

最终 Markdown 不交给模型来排版，而是由程序统一渲染。

固定格式如下：

```markdown
**问题明细**
**1. 问题标题**
问题定性：**高风险**

审查类型：XXX审查
原文位置：第X行；第X行
原文摘录：
- “……”
- “……”

风险判断：
……

法律/政策依据：
- 《……》
- 《……》
```

这样做有三个好处：

- 输出稳定
- 便于做批量处理
- 便于后续接入前端或数据库

## 10. 质量控制

建议加 3 个质检点。

### 10.1 召回质检

检查每份文件是否至少命中以下高频类中的一类：

- 证书
- 业绩
- 样品
- 检测报告
- 设备

如果一个都没命中，要记录 warning。

### 10.2 LLM 质检

检查 JSON 结构：

- 字段齐全
- 风险等级合法
- 行号格式合法
- 引文不能为空

### 10.3 输出质检

检查最终 Markdown：

- 是否有重复标题
- 是否有空的“法律依据”
- 是否有无法回溯原文的位置

## 11. 适配不同品类

这套方案不只适用于家具，可以通过“品类配置”扩展。

例如：

- 家具：证书、样品、检测报告、设备
- 医疗设备：进口、授权、注册证、售后响应、保修、临床参数
- 信息技术服务：人员证书、项目经理、驻场、本地化、著作权、业绩
- 物业管理：驻场人数、社保、本地库房、响应时限、管理体系

建议抽象为：

- 通用风险规则
- 品类特定规则
- 通用法条池
- 品类提示词增强

## 12. 推荐运行模式

### 12.1 开发调试模式

输出中间件：

- `normalized_txt/`
- `candidate_spans.json`
- `llm_raw/`
- `merged_findings.json`
- `final_md/`

这样便于检查问题是在解析、召回还是模型阶段发生的。

### 12.2 批量生产模式

只保留：

- 最终 Markdown
- 关键日志

## 14. 最小可用产品能力边界

第一版建议做到：

- 支持 `.docx/.doc/.txt`
- 支持按目录批处理
- 支持 OpenAI-compatible 接口调用 Qwen
- 支持按类别审查
- 支持输出标准 Markdown
- 支持保存中间调试文件

第一版不必强求：

- 全自动 OCR
- 深度语义检索
- 跨文件对比学习
- 复杂法条自动检索

## 15. 结论

要让 Qwen3.5-27B 做到接近人工审查效果，关键不是单纯“写更强 prompt”，而是把流程设计成：

- 规则高召回
- 模型低负载定性
- 输出结构化
- 程序统一渲染
- 结果再去重

这套方法比“整份文件直接丢给模型”稳定得多，也更容易批量扩展到不同品类。
